# Multi-Agent Smart Contract Vulnerability Detection: Experimental Results

This repository contains the raw evaluation data and environment configurations for our comparative study on smart contract vulnerability detection. We compared traditional static analysis tools against LLM-based tools, utilizing an n8n workflow for result aggregation and deduplication.

## 📁 Repository Structure
* `/Slither_Results/`: Raw output JSON/log files from Slither.
* `/SmartBugs_Results/`: Output from SmartBugs (including Solhint & Semgrep).
* `/GPTScan_Results/`: LLM-based analysis results targeting logic vulnerabilities.
* `/ChainGPT_Results/`: Audit reports generated by ChainGPT.
* `/Mythril_Results/`: Symbolic execution results.
* `/n8n_Aggregated_Summary/`: The final AI-summarized outputs, showing the deduplication process and false-positive reduction.

## 🛠️ Environment Configurations

### 1. Slither
* **Version:** 0.9.3
* **Python Environment:** Python 3.10
* **Solidity Compiler (solc):** 0.8.x (managed via solc-select)
* **Command used:** `slither . --json output.json`

### 2. SmartBugs (Solhint & Semgrep)
* **Version:** 2.0.15
* **Dependencies:** Docker Desktop (running), Python 3.13, py-cpuinfo
* **Command used:** 
```bash
./smartbugs -t all -f samples/* --timeout 600
./reparse results
./results2csv -p results > results.csv
```

### 3. GPTScan
We utilized the official [GPTScan repository](https://github.com/GPTScan/GPTScan.git) for detecting business logic vulnerabilities (S-class vulnerabilities). 

**Environment & Dependencies:**
* **Python Version:** Strictly `Python 3.10` (as required by the official repository for `falcon` and AST parsing compatibility).
* **Solidity Compiler:** Handled via `solc-select`.
* **Command used:**
  ```bash
  pip install -r requirements.txt
  solc-select install 0.8.19 # just an example of 0.8.19
  solc-select use 0.8.19
  python3.10 main.py -s /sourcecode -o /sourcecode/output.json -k OPEN_AI_API_KEY_xxxxxxxxxxxxx
  ```

## 4. ChainGPT & Mythril

- **ChainGPT:** Integrated through the ChainGPT Web API in the n8n workflow. Each Solidity contract is submitted via an HTTP `POST` request to `https://api.chaingpt.org/chat/stream` using bearer-token authentication. The request uses the `smart_contract_auditor` model and passes the Solidity source code as the audit query.

- **Mythril:** Deployed in a Docker-based local analysis environment and exposed as a self-hosted REST API. Each contract is analyzed via an HTTP `POST` request to `http://192.168.153.5000/api/v1/analyze`. The request payload includes the Solidity source code, the contract name, and runtime-control parameters such as `executionTimeout = 30`, `createTimeout = 15`, and `solverTimeout = 10000`.

## 5. AI Summary & Deduplication (n8n Workflow)

- **Platform:** n8n (local/cloud instance)

- **Core Logic:** Aggregates JSON outputs from ChainGPT, Mythril, and other analysis tools, then applies an LLM-based validation and summarization workflow to normalize vulnerability descriptions, cross-reference overlapping findings, remove duplicates, and filter likely false positives. The final output is a more concise and higher-confidence audit report with retained findings, explanations, and fix-oriented guidance.
